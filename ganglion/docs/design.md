# Ganglion 설계문서 (GECO1)

> 상태: **초안 / 하드웨어 소싱 중** (배선·설치 완료 예상 ≥ 2026-07-07)
> 대상 기기: **GECO1** — 데스크 미니 (pisound + rpi)
> 앱: **ganglion** — 헤드리스 인터페이스 (Qt 없음)
> 대비: GCaMP6s(본체) / synapse(터치스크린 앱)
> 갱신: 2026-07-04 — 시안 **2a**(상하 분할) 채택 + 실 1-bit 렌더/바이트 코스트 검증 반영.

이 문서는 코드 이전에 **정해야 할 설계 결정**을 모아 둔다. `[결정]`은 확정, `[열림]`은
사용자/시안 대기, `[조사]`는 리서치 결과. 인코더 UX 시안은 별도(사용자 준비).

## 0. 하드웨어 (확정 스펙)

| 요소 | 부품 | 인터페이스 | 비고 |
|---|---|---|---|
| 디스플레이 | Adafruit Monochrome **1.12" 128×128 OLED**, **SH1107**(=SSD1107 계열) | **I2C** (STEMMA QT) | 흑백 1bpp, 그레이스케일 없음 |
| 입력 ×2 | 로터리 **인코더 + 푸시스위치 + RGB LED** 통합 모듈 (Adafruit seesaw + NeoPixel) | **I2C** (seesaw) | 회전(상대)+누름+RGB 한 보드 |
| 오디오 | pisound HAT | — | 기존과 동일 |

> **7인치·풋스위치4·볼륨/exp 페달 없음.** GCaMP6s의 물리 I/O를 대부분 떼어낸 제약형 리그.

### I2C 주소 계획 `[결정]` — 실측 완료 (2026-07-16)
`i2cdetect -y 1` → `36 37 3d`. 셋 다 한 버스에 공존하고 충돌 없음.
- SH1107 디스플레이: **`0x3D`** — Adafruit 128×128 모듈은 `0x3D`로 스트랩되어 나온다.
  튜토리얼·예제가 죄다 `0x3C`라 코드도 `0x3C`로 적혀 있었고, 실물에서 안 잡혀서 발견됐다.
- seesaw 인코더: `0x36` / `0x37` (점퍼) — 커밋 288cdeb에서 검증됨.
- pisound와의 충돌 없음.

### 버스 속도 `[결정]` — 실측 (2026-07-16)
**버스는 100kHz로 출하돼 있었다.** `dtparam=i2c_arm=on`만 있고 `i2c_arm_baudrate`가 없으면
Pi 기본값이 100kHz다(device tree의 `i2c-1/of_node/clock-frequency`로 확인). 아래 §2 코스트
모델은 전부 **400kHz를 가정**하고 있었으므로 **published 수치는 4배 낙관적이었다.**
→ `config.txt`에 `dtparam=i2c_arm_baudrate=400000` 추가(재부팅 필요). SH1107 정격 내(fast mode).
`tools/oled_probe`는 device tree에서 실클럭을 읽어 billing하므로 이제 이 실수는 반복되지 않는다.

`scheduler.py` · `configs.py` · `taptempo.py` · `utils.py`

## 2. 디스플레이 성능 `[조사]` — 렌더링 아키텍처를 좌우하는 핵심

### 물리 한계 (측정 근거 기반)
- 프레임 크기: 128×128×1bpp = **2048 바이트/프레임**.
- **I2C 400kHz(fast mode):** 전체화면 **~10 fps(0/180° 방향)**, **~16 fps(90/270° 방향)**.
- **부분 갱신(1행 텍스트): ~5 ms.** ← 전체화면(60~100ms)보다 압도적으로 빠름.
- **I2C 1MHz 오버클럭(`dtparam=i2c_baudrate=1000000`):** ~2.5× 빨라지나 **SH1107 정격 초과 → 신뢰성 리스크**(글리치 시 폴백 필요).
- 참고: SPI 40MHz면 5~20ms지만 **이 보드는 I2C 전용(STEMMA QT)** — SPI 옵션 없음. 부드러운 애니메이션이 필수가 되면 SPI SH1107 보드로 교체해야 하는 아키텍처 제약.

### 방향 의존성의 이유
90/270°는 SH1107의 native page 구조에 프레임버퍼가 정렬돼 순차 쓰기가 가능, 0/180°는
column 단위로 흩어져 씀. 방향을 고를 수 있었다면 90°가 공짜로 16fps였을 것이다.

### 설계 결정
- `[결정]` **부분/더티-리전 렌더링을 기본으로.** 전체화면 재그리기는 "비싼 연산"으로 취급.
  매 틱 full redraw 금지 — 바뀐 영역만 push.
- `[결정]` **장착 방향 = luma `rotate=0`** — 실기 확정 (2026-07-16, `tools/oled_probe --stage orient`).
  이전의 "FFC-down 장착 = 180° = `rotate=2`" (2026-07-12 잠정)는 **연역이 틀렸다**: 실물은 회전
  없이 똑바로 뜬다. 성능은 동일하므로(0/180° 모두 SH1107의 느린 축) 이 오류는 유리를 보기
  전까지 드러날 방법이 없었다 — 부품 도착 전 확정한 값의 전형적인 실패 방식이다.
  "90° 회전 장착 전제(공짜 16fps)"는 여전히 **폐기** — FFC-down 부동산 제약이 방향을 강제하므로
  성능을 이유로 방향을 고를 수 없다. 따라서 **풀프레임은 느린 레짐이 천장**이고, 위의 더티-리전
  규율은 최적화가 아니라 **전제조건**이 된다.
- `[결정]` **1bpp 흑백 UI 문법:** 그레이스케일 없음 → 볼드 라인아트·큰 텍스트·아이콘·반전(invert)
  블록으로 정보 표현. 안티에일리어싱 폰트 못 씀 → 비트맵 폰트.
- `[결정]` **폰트 = Silkscreen(본문 8/16/24/32) + Micro5(플로어 티어).** 실 1-bit 렌더로 검증:
  Silkscreen 전 티어 또렷. **Micro5는 @6/@8 뭉갬 → @10pt에서 legible**("5px" 폰트라 5px 글리프를
  내려면 10pt 래스터 필요). 시안의 "6px→6pt" 매핑은 우리 PIL/luma 경로에서 오류였음 → **Micro5@10 고정.**
  OFL 번들: `ganglion/assets/fonts/`.
- `[결정]` **실시간 요소 프레임 예산:** 작은 영역만 갱신하면 부분갱신으로 충분. 아래 코스트 모델로 정량 확인됨
  — 실시간 위젯은 화면의 좁은 밴드에 격리.

### 드라이버 선택 `[열림 → 조사]`
- **1순위: `luma.oled`** — Pillow(PIL) 네이티브 드로잉. 우리가 UI를 직접 그리는 방식에 가장 잘 맞음.
  설치 시 실제 sh1107 디바이스 지원 버전 확정 필요.
- **2순위: `adafruit-circuitpython-displayio-sh1107`** (+Blinka) — displayio 위젯 지향, 커스텀 드로잉엔 덜 편함.
- 결정 기준: 커스텀 UI + 부분갱신 제어권 → **luma.oled 우선**. 벤치 실측 후 확정.

### 측정: 바이트 코스트 모델 `[조사]` — `ganglion/i2c_cost.py`
SH1107은 **페이지=8px×128** 단위 저장(16 페이지행). 부분갱신 비용 = *(건드린 페이지행 수 × 폭)* + 명령 오버헤드.
시안 2a 화면을 실 1-bit로 렌더(`ganglion/design_screens.py`)하고 프레임 diff로 실측:

| 리프레시 이벤트 | 전송 | @400kHz | 성격 |
|---|---|---|---|
| 전체 재그리기 | 2192 B | 49ms → 20fps* | 화면 전환 시에만 |
| 노드 스크롤(스트립+밴드) | 1379 B | 31ms → 32fps | 이산 조작 |
| 스냅샷 전환 | 377 B | 8.5ms → 118fps | 저렴 |
| 마퀴 1스텝(이름 밴드) | 349 B | 7.8ms → 127fps | 저렴 |
| 튜너 바늘 1스텝 | 155 B | 3.5ms → 287fps | 저렴 |
| 노브 값 1개 | 37 B | 0.8ms → 1201fps | 무시 가능 |

*(전체 20fps는 와이어타임 상한 — 실측은 SW·방향 오버헤드 포함 **~10fps**(180° 장착 = 느린 레짐))*

- `[결정]` **핵심:** 비싼 건 화면 **전체가 바뀌는 순간**(전환·노드스크롤)뿐이고 이건 *이산 조작*. **마퀴·바늘·미터·노브
  같은 매 프레임 애니메이션은 한 페이지 밴드만 건드려 전부 저렴**(이전 "마퀴가 리스크" 판단은 코스트 모델로 반증됨).
- `[결정]` 규율: 정적 레이아웃은 표현력 마음껏 / 매 프레임 애니메이션은 **한 페이지 밴드에 격리** → 항상 두 자릿수 fps.

### 실측: diff 드라이버 `[결정]` — `ganglion/hw/oled.py`, 벤치 `tools/oled_bench.py`
위 표는 *시안* 화면의 손으로 고른 before/after 쌍이었다. 이건 **실제 `app.render` 프레임 시퀀스**를
실 틱레이트(33Hz)로 돌려 실 드라이버(`DiffSink`)에 밀어넣은 값 — 패널이 진짜 보게 될 숫자다.

| 시나리오 | 평균/틱 | 최악 | 버스 점유(30ms 틱) | 무전송 틱 |
|---|---|---|---|---|
| 풀프레임 싱크(구 스텁) | 2192 B / **49 ms** | — | **163% — 틱보다 길다** | 0% |
| chain 휴지(16px 노드 마퀴) | 116 B / 2.6 ms | 4.5 ms | 8.7% | 41% |
| glance 휴지(보드 마퀴) | 173 B / 3.9 ms | 8.2 ms | 12.9% | 49% |
| glance, 이름 둘 다 스크롤 | 278 B / 6.3 ms | 13.6 ms | 20.9% | 49% |
| 튜너(바늘 + cents 표시) | 34 B / 0.8 ms | 3.4 ms | 2.6% | 26% |

- `[결정]` **풀프레임 싱크 폐기 → `DiffSink` 채택.** 풀프레임 49ms는 30ms 틱보다 **길다** — 방향 페널티
  이전에 이미 인코더 폴링(같은 I2C 버스)을 굶긴다. diff는 (1) 무변화 틱엔 **1바이트도 안 보내고**,
  (2) 바뀐 페이지행만 패킹하며(bbox는 PIL C레벨), (3) 행마다 바뀐 컬럼 스팬만 밀어넣는다.
  **무손실 검증됨**: 보낸 스팬만 재생한 가짜 패널의 프레임버퍼가 매 틱 원본 프레임과 바이트 단위로 일치
  (페이지 누락 = 조용한 화면 깨짐이라 벤치가 매번 확인한다).
- `[결정]` **튜너 미터는 가로 유지.** 세로축 변형을 같은 cents 트랙으로 측정 → 평균 30 B vs 가로 34 B,
  최악은 오히려 174 B vs 153 B. 기하가 상쇄된다(세로 막대×가로 이동 = 3페이지행×좁은 스팬 ↔ 가로 막대×세로
  이동 = 1~2페이지행×21컬럼 고정폭). 30ms 틱에서 둘 다 1ms 미만 → **성능은 이 선택의 근거가 못 된다.**
- `[결정]` **glance 스냅샷 이름 = 16px 티어**(보드 이름만 24px). 24px 이름 **둘이 동시에** 흐르면
  7.0ms/틱(버스 23%)으로, 앱의 다른 모든 애니메이션을 합친 것보다 비쌌다 → 위 규율의 실제 위반 지점.
  한 티어 내리니 흔한 이름은 아예 안 넘쳐 **마퀴가 멈추고**(3.9ms), 긴 생성 이름이 걸려도 밴드가
  3→2 페이지행이라 싸다. 위계상으로도 맞다(보드가 주어, 스냅샷이 수식어).

### 실기 확정 `[결정]` — 패널 도착 후 (2026-07-16), `tools/oled_probe.py`
위 두 절은 전부 *가짜 패널* 위의 숫자였다. 실물에서 확인/수정된 것:

- `[결정]` **`LumaWriter` 검증 완료.** 스텁이 달고 있던 미지수 3개가 전부 우리 쪽에 유리하게 풀렸다.
  (1) **어드레싱 모드** — luma의 sh1107 init은 맨 `0x20`을 쏜다. luma가 SSD1306 상수표를 이 칩에
  재사용하는 탓에 그건 SSD1306의 *2바이트* MEMORYMODE opcode지만, **SH1107에선 `0x20`이 "페이지
  어드레싱"을 뜻하는 완결된 1바이트 명령**이다(`0x21`이 vertical). 즉 우리 명령이 가정하던 모드에
  칩이 이미 들어와 있다 — 걱정했던 건 정반대 경우였다. (2) **컬럼 오프셋 0.** (3) **페이지 축 전치 없음.**
- `[측정]` **풀프레임 ≈ 220ms** (100kHz 기준. 184ms가 와이어, 나머지는 luma의 픽셀 단위 파이썬
  패킹 루프). 400kHz에서도 ~80ms — **여전히 틱의 3배**다. diff 드라이버는 최적화가 아니라 생존조건.
  diff 틱 ~10ms, 무변화 틱 **0.4ms**.
- `[주의]` **`dev.persist = True`** 없으면 luma의 atexit 훅이 패널을 지우고 꺼버린다. 앱엔 맞지만
  테스트 화면엔 치명적.

### 전력 / 플리커 상한 `[결정]` — `tools/oled_probe --stage power`
패널엔 텔레메트리가 없지만 3V3 레일에 물려 있고 Pi 5 PMIC에 ADC가 있다 → **패널 상태별 차분**으로
측정된다(60샘플 중앙값, 분해능 ±2mA). `3V3_SYS_A` 증분:

| 상태 | 증분 |
|---|---|
| 패널 on, 전부 검정 | +7 mA |
| **실제 UI(15% 점등)** | **+44 mA** |
| 25% 점등 | +83 mA |
| 50% 점등 | +72 mA |
| 100% 점등 | +79 mA |
| 100% @contrast 0x01 / 0x40 / 0x7F / 0xFF | +49 / +63 / +78 / +87 mA |

- `[결정]` **전류는 점등 픽셀 *수*를 따라가지 않는다 — 따라가는 건 "한 열에 몇 개냐"다.**
  같은 4096px을 배치만 바꿔 측정(`--stage scanline`): 세로 32px 폭 막대(128/col) **9.3mA·플리커**
  vs 가로 막대(32/col) **83mA·정상** vs 흩뿌림(32/col) **80mA·정상**. 점등 수가 동일한데 전류가
  **9배 붕괴**한다 — 발광이 끊기니 평균 전류가 무너지는 것. **낮은 수치는 절약이 아니라 실패 신호.**
  패시브 매트릭스는 한 스캔라인을 켤 때 그 줄의 점등 세그먼트가 *동시에* charge pump를 빤다
  → **패미컴의 스캔라인당 스프라이트 제한과 같은 구조**(사용자 통찰). 우리 이미지 기준 스캔축은
  **세로**. 따라서 **긴 세로 런**이 유일한 유발 조건이고, **같은 잉크를 흩뿌리면(디더) 사라진다.**
- `[폐기]` ~~"면적이 넓을수록 브라운아웃"~~ — 변수를 **총 면적**으로 잘못 짚었던 초기 진단.
  실제 변수는 **열당 연속 길이**다. `scattered50`(128/col)이 8192px에 18.5mA로 "효율적"이던 게
  사실은 붕괴였다. `[열림]` 전백(모든 열 128/col)이 이 모델대로면 붕괴해야 하는데 76.6mA로
  높게 나온다 — 미해결.
- `[폐기]` ~~contrast가 무릎을 옮기는 레버~~ — **약한 레버였다.** 전 범위(0x01→0xFF)가 전류
  1.8배(49→87mA)뿐이고, 밝기 지각은 로그/멱함수라 체감 ~20%다. 그래서 **모든 밝기에서 플리커가
  동일**했고 사용자는 밝기 변화조차 잘 못 느꼈다. `SYSTEM > Brightness`는 플리커 대책이 아니라
  그냥 취향 설정이다.
- `[폐기]` ~~리프레시 레이트 가설~~ — luma 기본 `0xD5=0x80`은 128-mux에서 ~60Hz라 Granit-Harper
  (CFF∝log면적)로 설명될 뻔했으나, **발진주파수를 0x0~0xF로 스윕해도 플리커가 전혀 안 변했다**
  (`--stage refresh`). 당연하다 — 프레임레이트는 *열당 부하*를 안 바꾼다. 반증됨.
- `[결정]` **육안으로는 실사용 문제가 아니다** (사용자 판정). 헤드룸도 충분(3V3 정격 0.5~0.6A 중
  OLED 최대 ~0.1A). → **UI 제약을 걸지 않는다.** 아래 "1bpp 문법 상한"은 폐기. 만약 나중에
  전면 반전 같은 게 필요해지면 **디더가 예비 레버**로 남아 있다(위 diag 80mA 정상).
- `[결정]` **급전은 용의선상에서 제외.** 재론 방지용으로 근거를 남긴다:
  Pi 3V3 레일은 최악(전백@0xFF)에서도 **총 200mA / 3.2887V로 미동 없음**(ADC가 상수를 뱉는 게
  아님은 확인 — 다른 시점 3.28776V). DA9091 PMIC의 3.3V 공급능력은 600mA(CM5 데이터시트)라 **정격의
  1/3**이다. 흔히 인용되는 "50mA"는 **GPIO 출력 핀 합계**지 3V3 전원 핀이 아니다(공식 문서는 3V3
  핀 수치를 아예 싣지 않는다). 배선도 아니다 — OLED는 데이지체인 **맨 앞**이고 그 앞단은 pisound
  스루홀 헤더뿐.
  → **패널 보드 자체의 구동 출력 한계**로 확정(SH1107 내부 charge pump가 3.3V→~7~9V 부스트;
  주저앉는 지점이 부스터 출력단이라 Pi의 3.3V엔 나타날 수가 없다). **더 센 전원·굵은 배선·버스
  속도 어느 것도 안 고친다.** 레버는 contrast와 점등 면적 둘뿐.
- `[결정]` **1bpp UI 문법은 그대로 간다.** 한때 "큰 반전 블록 금지"를 여기 적었으나 **폐기** —
  육안 판정상 실사용 문제가 아니고, 유발 조건도 "큰 면적"이 아니라 "긴 세로 런"이었다. 현 UI는
  15% 점등에 세로 런도 짧아 무관. 이 절은 **전력 예산의 근거**로만 남긴다(아래).
- `[결정]` **전력 예산 — 문제없음.** 3V3 레일 총계(실측, PMIC 차분):
  Pi 자체 ~122mA + OLED 최대 ~80mA(전백@0xFF) + NeoPixel 2개 풀화이트 ~58mA = **최악 ~260mA**.
  정격 600mA(DA9091, CM5 데이터시트) 대비 **절반 이하** → 마진이 커서 재론 불필요.
  LED가 OLED에 필적하긴 하나(58 vs 80) 둘 다 여유 안. 참고: 흔히 인용되는 "50mA"는 GPIO *출력
  핀* 합계지 3V3 전원 핀이 아니다(§0).
- `[결정]` **NeoPixel 밝기 = 0.1** (`hw/seesaw.py _LED_BRIGHTNESS`), 실기 육안 확정.
  풀스케일은 물론 0.5도 눈앞에 두기엔 밝았다. **전력이 이유가 아니다**(2개 풀화이트 58mA,
  예산 340mA 여유) — 순전히 편의. `LED_RGB`를 깎지 않고 여기서 스케일하는 이유: 그 맵은 색의
  *의미*(decision I)를 담고 밝기는 담지 않는다. 섞으면 팔레트 수정 때마다 dim을 재유도해야 한다.
- `[결정]` **팔레트 6색은 0.1에서도 서로 구분된다** — 정상품(0x37) 육안 확인. WS2812는 전역 전류
  제어가 없어 `brightness`가 8비트 값을 곱할 뿐이라, 0.1이면 채널당 2~26으로 눌린다
  (grey→(6,6,6), blue→(0,9,25) vs purple→(15,0,25)로 B가 지배). 이 해상도에서도 통과했으므로
  **decision I의 색 의미론은 실기에서 성립**. 밝기를 더 내리면 여기가 먼저 깨진다 → 그때는
  팔레트가 아니라 `_LED_BRIGHTNESS`를 의심할 것.
- `[참고]` 색상별 전류(밝기 1.0, 2개 합산): amber 24.9 / green 26.4 / blue 27.8 / red 27.3 /
  purple 35.1 / grey 13.2 mA. purple이 30~40% 높으나 **0.1에서 무의미**해졌고, 애초에 이 측정은
  0x36 보드 파손 상태에서 잰 것이라 신뢰도가 낮다(인코더 디솔더-교체 중 파손, 교체 예정).

### 유휴 전력 / 번인 방어 `[결정]` — 라이브 서비스 실측 (2026-07-17)
위 표들은 전부 *테스트 화면*의 수치다. 이건 **라이브 systemd 서비스**를 재시작해 idle을 0에 맞추고
실 UI가 스스로 dim/off로 넘어가는 것을 3V3 레일에서 잰 값 — 앱이 실제로 하는 일이다.
구현은 `hw/oled.PanelPower` + `Runtime` (30초 dim / 5분 off, [`decisions.md`](decisions.md) S).

| 상태 | 레일 총계 | 차분 | 패널 몫 |
|---|---|---|---|
| active (`contrast 0x7F`) | 165.9 mA | — | 43.9 mA |
| **dim** (`contrast 0x01`, 30s) | 134.2 mA | **−31.7** | 12.2 mA (**−72%**) |
| **off** (`hide()`/`0xAE`, 5min) | 122.0 mA | **−43.9** | **~0** |

- `[결정]` **off는 패널을 진짜 0으로 만든다** — 122.0mA는 위 "전력 예산"에서 따로 잰 **Pi 자체
  소비 ~122mA와 같은 값**이다. 즉 `0xAE` 뒤엔 패널 몫이 남지 않는다. active의 패널 몫 43.9mA도
  위 표의 "실제 UI(15% 점등) +44mA"와 독립적으로 일치 → **두 측정이 서로 교차검증**된다.
- `[결정]` **contrast의 세기는 화면 내용에 의존한다.** 위의 `[폐기]`("contrast는 약한 레버")는
  **전백에서 참**이고(0x01/0x7F = 49/78mA, −37%), **우리 UI에선 거짓**이다(43.9→12.2mA, **−72%**).
  §2 자체 모델과 정합한다: 전백은 charge pump가 포화된 레짐이라 더 깎을 게 없고, 15% 점등·짧은
  세로 런인 실 UI엔 헤드룸이 있어 거의 선형으로 듣는다. → **전백 수치로 일반화하지 말 것.**
  (`decisions.md` S를 처음 적을 때 전백에서 "−44%"를 연역했다가 실측에 반증된 자리다.)
- `[결정]` **`SYSTEM > Brightness`의 성격은 그대로 취향 설정**이다 — 이 발견은 *전류*가 화면 내용에
  의존한다는 것이지, 사용자가 **밝기 변화를 지각한다**는 뜻이 아니다(지각은 여전히 로그/멱함수라
  ~20%). 위 `[폐기]` 두 줄은 유지된다. 바뀐 건 **번인 맥락에서 contrast가 쓸모 있다**는 것뿐.
- `[결정]` **`0xAE`는 GDDRAM을 보존한다 — 실물 확인**(hide→show 후 프레임 그대로). blank 중
  `_draw()`를 건너뛰는 최적화가 이 사실 위에 서 있다: 지워졌다면 앱 상태가 그대론 한 diff가
  0바이트라 **빈 화면으로 깨어난다**. 데이터시트로 연역하고 넘어갈 자리가 아니었다.

## 3. 입력 모델 `[결정]` — 시안 2a 채택

물리 입력: **로터리 2개(회전 상대델타) + 푸시 2개 + RGB LED 2개**. 시안 2a가 이걸 다음으로 확정:

- `[결정]` **상하 밴드 매핑:** 인코더가 화면 좌측에 **세로로** 배치 → **ENC0(위)=상단 밴드(구조/노드체인),
  ENC1(아래)=하단 밴드(값/노브)**. 물리 위치 = 조작 영역, 학습 없이 직관적.
- `[결정]` **입력 어휘 = 우리 `GestureRecognizer`와 정확히 일치:** Rotate + Press{click,long} + Combo.
  ENC0 CLICK=슬롯 메뉴(바이패스·이동·교체·제거), ENC0 HOLD=한 단계 위(→깊이 −1), ENC1 CLICK=노브 락(점선→실선),
  ENC1 HOLD=튜너, **0+1 COMBO=스냅샷 저장**. → 하드웨어 확장 불필요.
- `[결정]` **회전은 상대 델타 + 가속.** (구현됨: `ganglion/input.py`)
- `[결정]` **RGB LED 의미론** (시안 `leds()` 채택): ENC0=이펙트 카테고리 버킷색(Drive=주황·EQ=파랑·Mod=보라…),
  바이패스=적색 / ENC1=락·편집=녹색, 메뉴=주황, 삭제=적색, **튜너=보라**, 이동=파랑. → **열린결정 #3 닫힘.**
- `[결정]` **폴링 확정. seesaw INT는 조건부 백로그.** 커뮤니티 정찰(2026-07-07): seesaw INT는
  레벨 신호 + **read-to-clear**(위치를 I2C로 읽어야 플래그 클리어) → 어차피 읽기 1회 필수라 오프로드
  아님. "폴링 없이 INT가 안 뜬다"·"멀티 인코더 INT wired-OR 시 불안정(~2.6V 처짐)"·쿼드러처 카운트
  드리프트(seesaw#51, open) 리포트 다수. 노브는 사람 손 속도라 33Hz 폴링으로 충분하고, 버스 경합의
  주범은 인코더 폴이 아니라 OLED 풀프레임. → 현 스텁(`ganglion/hw/seesaw.py`)의 폴링 유지. INT는
  실기 벤치에서 "OLED 전송이 인코더 폴을 굶겨 입력 레이턴시가 튄다"가 실측될 때만, 모듈당 별도 GPIO +
  read-to-clear 패턴으로 재검토.

## 9. 열린 결정 요약

**하드웨어 도착 후 실측:**
1. ~~인코더 0x36/0x37 확인(Pi 5)~~ · ~~디스플레이 주소·pisound 충돌~~ — `i2cdetect` → `36 37 3d`,
   충돌 없음. 디스플레이는 **`0x3D`**(코드의 `0x3C`는 틀렸었다). §0 참조
2. ~~폴링 확정~~ (seesaw INT 미배선; 33Hz 폴 검증됨)
3. ~~회전 부호·스위치/픽셀 핀 확정~~ — 온메탈 검증: 부호 반대라 `invert=(True,True)` 기본,
   핀 24/6 동작. seesaw 간헐 NACK(Errno 121)은 `_retry_io`+색 dedup으로 처리 (`hw/seesaw.py`)
4. ~~luma.oled sh1107 지원 버전 확정~~ — **3.15.0** (128×128 지원은 3.11.0+). `LumaWriter`
   온메탈 검증 완료, 장착 방향 **`rotate=0`** 확정. §2 "실기 확정" 참조
5. ~~버스 속도~~ — 100kHz로 출하돼 있던 걸 발견(§0). `dtparam=i2c_arm_baudrate=400000` 적용.
   **1MHz 오버클럭은 불필요해 보인다**: 남은 병목은 대역이 아니라 패널 구동 전류(§2 플리커)
6. Micro5@10 실 패널 재확인 (UI는 떴고 티어 legibility는 육안 확인 남음)
7. 플리커가 멎는 contrast 실측 → `SYSTEM > Brightness` 상한으로 고정 (§2)

**설계/구현 진행 중 결정:**
8. ~~부팅 자동실행 방식~~ — **systemd 확정**: `deploy/ganglion-service/`
   (`app.py --device --live`). 핵심은 `KillSignal=SIGINT` — 파이썬 기본 SIGTERM은
   `finally`/`atexit`를 안 태워서 stop 때마다 OLED·LED가 켜진 채 남는다(번인).
9. GECO 자체 MIDI 정체성 필요 여부
10. 헤더 이름 티어 (24px+마퀴 vs 16px)
11. 플러그인 화이트리스트 확정본 여부 (시안 `WL` ↔ `plugincatalog.py` 실 LV2 URI 매핑)
12. ~~`10초 무조작 → 깊이 −1 자동 진입`(스크린세이버성) 유지 여부~~ — **폐기 → 번인 방어로 대체**
    (2026-07-17). 그 항목은 부품이 없던 시절 "무조작 상태를 뭐에 쓸까"에서 나온 **기능성 탐구
    설계**였다. 실물을 단 지금 무조작 상태의 임자는 **패널 수명**이고, 화면을 끌 거면 그 밑에서
    depth를 바꿔 두는 건 의미가 없다(깨어나 맥락만 잃는다). → **30초 dim / 5분 off**,
    [`decisions.md`](decisions.md) S · 아래 §2 "유휴 전력".

## 관련 코드 (ganglion/)
- `display.py` — 128×128 프레임버퍼 + 터미널 렌더러(braille/half)
- `input.py` — `GestureRecognizer`(공유) + 키보드 소스. Rotate/Press{click,long}/Combo
- `runtime.py` — 헤드리스 메인루프(결정 J). 입력 폴·뷰·스플래시(F)·LED·**유휴 패널 전력**(S)
- `hw/seesaw.py` — 실물 인코더 입력 (온메탈 검증; `invert` 기본 True, `_retry_io`+색 dedup)
- `hw/oled.py` — 실물 SH1107 출력. `DiffSink`(바뀐 스팬만) + `LumaWriter`(와이어) +
  `PanelPower`(30s dim / 5min off, 번인 방어 — 결정 S)
- `tools/encoder_bench.py` — 인코더 온메탈 브링업(enc0=hue, enc1=밝기, 두 LED 미러)
- `tools/oled_bench.py` — 실 프레임 시퀀스로 diff 드라이버 무손실 검증 + 바이트 실측 (§2)
- `tools/oled_probe.py` — 실 패널 계측(방향·diff·전력·스캔라인·리프레시). PMIC ADC 차분 (§2)
- `emulator.py` — 하드웨어 없이 화면·인터랙션 검증하는 CLI
- `i2c_cost.py` — SH1107 바이트 코스트 모델 (§2 측정)
- `design_screens.py` — 시안 2a 화면(glance/chain/tuner) 실 1-bit 포팅 + 렌더/코스트 데모
- `assets/fonts/` — Silkscreen R/B + Micro5 (OFL 번들)
